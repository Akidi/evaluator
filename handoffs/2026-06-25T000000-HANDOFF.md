## Where we're going from here

User flagged the **evaluator** as the likely next focus. There's no `evaluator/` (or similar) directory yet ‚Äî `parser/types.ts` defines the `Node` union the evaluator will walk:

```
NumNode    { type: 'Num', value: number }
IdentNode  { type: 'Ident', name: string }
UnaryNode  { type: 'Unary', op: 'MINUS' | 'NOT', operand: Node }
BinaryNode { type: 'Binary', op: Kind, left: Node, right: Node }
CallNode   { type: 'Call', callee: Node, args: Node[] }
TernaryNode{ type: 'Ternary', test: Node, ifTrue: Node, ifFalse: Node }
```

Concrete next steps (concept-only until the user implements):

- Decide how `IdentNode` resolves to a value ‚Äî needs an environment/context (variable bindings) passed into the evaluator, since `Ident` nodes currently just carry a bare `name` string.
- Decide how `CallNode` resolves `callee` to an actual function ‚Äî likely a registry of builtin functions (`max`, `pow`, `abs` appear in existing parser tests as stand-ins) keyed by name, since `callee` is an `IdentNode`, not a function reference.
- Decide `BinaryNode`'s `op: Kind` dispatch ‚Äî it's the raw lexer `Kind` (`"PLUS"`, `"AND"`, `"EQ"`, etc.), so the evaluator needs a switch/map from `Kind` to an actual operation (arithmetic, logical short-circuiting for `AND`/`OR`?, comparison).
- Decide return type: does the evaluator return a single `number | boolean | string`, or a tagged union? Comparisons/`NOT`/`AND`/`OR` naturally produce booleans, arithmetic produces numbers ‚Äî first design decision before writing any evaluator code.
- Mirror the existing test style (`parser.test.ts`'s `describe`/`it` + direct `lexer.scan` ‚Üí `parser.parse` ‚Üí assert pattern) once an `evaluator.test.ts` exists.

## Open questions (not yet decided)

- **What does `AND`/`OR` evaluate to, and is it short-circuiting?** If `Ident`/`Call` operands can have side effects (unlikely for a formula language, but worth deciding explicitly), short-circuit evaluation matters. If purely value-based, order doesn't matter functionally but still affects whether both sides get evaluated/type-checked.
- **What's the evaluation context shape?** (e.g. a flat `Record<string, number>` for idents, vs. something richer that also holds the function registry.) This shapes the evaluator's function signature before any code gets written.
- **Error handling for undefined idents/calls at eval time** ‚Äî parse-time errors already exist (`ParserUnexpectedTokenError`, etc. in `parser/errors.ts`); does the evaluator get its own error hierarchy (e.g. `shared/errors.ts` already exists as a shared base ‚Äî worth checking if it's meant to be extended here)?
