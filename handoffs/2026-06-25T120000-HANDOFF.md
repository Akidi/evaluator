## Evaluator design decisions/leanings from this session (no code written)

- **No `raw`/lexeme field needed on AST nodes.** Tokens already have `position: {start, end, line}`; that plus `cursor.getSource()` is enough for diagnostics. The evaluator should consume the already-typed values (`Num.value: number`, `Ident.name: string`), not re-derive from source text.
- **`BinaryNode.op` dispatch:** a lookup table/switch mapping `Kind` (`"PLUS"`, `"EQ"`, etc.) → operation, mirroring how `Parser.INFIX_BP`/`PREFIX_BP` already map `Kind` → binding power. `AND`/`OR` likely can't live in a flat map if short-circuiting matters — a plain table evaluates both operands eagerly before dispatch, so short-circuit logic needs a `switch`/special-case instead.
- **Evaluator shape:** `switch(node.type)` over the `Node` union, one case per variant (`Num`, `Ident`, `Unary`, `Binary`, `Call`, `Ternary`), `default: throw` (ideally with a `never`-typed assignment for compile-time exhaustiveness, mirroring `parsePrimary`'s `default: throw new ParserUnexpectedTokenError(...)` pattern).
- **Leaf vs. branch is just the discriminated union, not a depth check.** `Num`/`Ident` return directly (no recursion); `Unary`/`Binary`/`Call`/`Ternary` recurse into their own child fields. Tree shape/depth asymmetry between branches is irrelevant — each recursive call only cares about its own subtree.
- **`Ternary` is the one case that must NOT eagerly evaluate both children** — evaluate `test` first, then evaluate only `ifTrue` or only `ifFalse` based on it. This is the short-circuit-by-construction case.
- **Var env (v1): flat `Record<string, number>`.** Vars resolving to other formulas (lazy/derived values) is explicitly **deferred** as a separate future decision — it introduces cycle-detection concerns (`A = B + 1`, `B = A * 2`) that shouldn't be bundled into the first working evaluator.
- **Missing-ident error strategy: walk-and-throw-when-hit**, not a separate pre-validation pass that batch-collects all missing vars. Consistent with the parser's existing fail-fast style.
- **Rejected: checking idents at the lexer level.** The lexer can't disambiguate a variable reference from a function name — `IDENT` is the same token kind for `x` and `max`; only the parser knows (by checking whether `LPAREN` immediately follows) whether an ident becomes a `CallNode.callee` or stays an `IdentNode`.

## Where we're going from here — DEFINITIVE NEXT TASK

**Build ident-checking into the parser itself, collected during the existing parse traversal (not a second tree walk).**

- Every time `parsePrimary` builds an `IdentNode`, check `this.peek().kind === 'LPAREN'` (the same check already implicitly needed to decide whether this ident is about to become a `CallNode.callee` via the `LPAREN` branch in `parseExpr`'s infix loop).
- Push a tagged entry to a list: `{ name: string, type: 'FN' | 'VAR' }` — `'FN'` if a call follows, `'VAR'` otherwise. This replaces the earlier, messier idea of collecting all idents and filtering out callees after the fact.
- This list is "free" — O(1) marginal cost per ident, piggybacking on the traversal the parser is already doing, no separate pass over the AST or token stream.
- **Open question to resolve as part of implementing this:** how is the list exposed from `Parser`'s public contract? `Parser.parse()` currently returns just a `Node` ([parser/types.ts](parser/types.ts) `IParser` interface). Options: change `parse()`'s return type to include the list, expose it as a public field set as a side effect of `parse()`, or something else. Decide deliberately, don't let it fall out as an implicit side effect.
- Once this exists, it directly feeds the evaluator's var/function validation: `'VAR'` entries get checked against the var env (`Record<string, number>`), `'FN'` entries get checked against the function registry — two different existence checks against two different maps, rather than one undifferentiated list.

## Open questions (carried forward, not yet decided)

- Does `AND`/`OR` evaluation need to be short-circuiting in practice (depends on whether `Ident`/`Call` operands can have side effects — probably not for a pure formula language, but not yet explicitly ruled out).
- Evaluator's error hierarchy — does it get its own (mirroring `parser/errors.ts`), or extend `shared/errors.ts::ProjectError`?
- Return type of the evaluator: single `number | boolean | string` vs. a tagged union (arithmetic → number, comparisons/`NOT`/`AND`/`OR` → boolean).
- The two parser gaps (missing `PERCENT` binding power, missing EOF check in `parse()`) — flagged, not fixed.
