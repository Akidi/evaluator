## State of `evaluator/evaluator.ts` (verify current state before continuing — user edits between turns)

- `Num`, `Ident` (with `setVar`/`UndefinedVariableError`), `Unary` (`MINUS`/`NOT`, evaluates operand first then checks `typeof` result — NOT AST `node.type` — to decide numeric vs boolean), and `Binary` arithmetic (`PLUS`, `STAR` confirmed; `MINUS`/`SLASH`/`PERCENT`/`CARROT` follow the same `MATH_OPS` table pattern but not yet added) are implemented and green.
- `Binary` case structure: `MATH_OPS: Map<Kind, MathFn>` + `COMP_OPS: Map<Kind, CompareFn>` (currently empty) registries, both keyed by lexer `Kind` (not `string`) since binary ops are a closed token set. `evaluate()` evaluates `left`/`right` first, looks up the op in `MATH_OPS`, and throws a Binary-specific `EvaluatorError` (not the generic "Unhandled node type") if the op isn't found in either table yet — this was a deliberate fix this session (see below).
- `MathFn`/`CompareFn` types: discussed but not yet written into `evaluator/types.ts` as of last check — recommended shape was a shared generic `type BinaryOpFn<R> = (a: number, b: number) => R` to avoid duplicating the `(a: number, b: number) => ...` signature twice. **Check `evaluator/types.ts` current state — user may have added this already.**
- `AND`/`OR` are NOT implemented yet and deliberately excluded from `COMP_OPS` — user confirmed they want short-circuit evaluation, which means special-cased branches in `evaluate()` (can't use a lookup-table-of-already-evaluated-operands approach for these, since the whole point of short-circuit is *not* evaluating `right` unconditionally). This resolves the previously-open "do AND/OR need short-circuit" question from prior handoffs: **yes, short-circuit, special-cased.**
- Bug fixed this session: `case "Binary":` originally had no block scope (`{ }`) around its `const` declarations — harmless only because it was the last case, but would've broken the moment another case got added after it. Now wrapped in braces. Variable naming also fixed: was `op`/`operand` for the evaluated left/right values (`op` reads like "the operator" but wasn't), renamed to `left`/`right` to match the AST field names and avoid confusion with `ast.op` (the actual operator).
- A "no leakage" test was added (`evaluator.test.ts`) asserting that an unimplemented `Binary` op (e.g. `AND`) throws the new Binary-specific `EvaluatorError`, NOT the generic `"Unhandled node type"` fallback — proves the scoping fix and specific-error fix both work.
